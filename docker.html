01 Foundation

What is Docker?

Docker is an open-source containerization platform that allows developers to package applications and their dependencies into lightweight, portable units called containers. Once packaged, a container runs identically on any machine that has Docker installed — whether it is a developer's laptop, a staging server, or a production cloud instance.

"Works on my machine" becomes works everywhere. Docker wraps your app, runtime, libraries, and config into one self-contained unit.

Before Docker, deploying software meant dealing with dependency hell — the same code behaved differently across environments because of mismatched OS versions, library conflicts, and configuration drift. Docker solves this by isolating each application in its own environment.

Why Developers Love It

  • No more "works on my machine" problems
  • Same image runs in dev, staging, and production
  • Faster startup than full Virtual Machines
  • Efficient resource usage — shares OS kernel
  • Version control for your entire runtime
  • One-command deploys with Docker Hub

Honest Tradeoffs

  • Adds complexity to simple single-app setups
  • Requires Linux kernel (VMs needed on Mac/Windows)
  • Stateful apps need careful volume management
  • Learning curve for networking and security
  • Container sprawl if not managed well
02 Architecture

Docker Architecture

Docker uses a client-server architecture. The Docker Client (your terminal) communicates with the Docker Daemon over a REST API. The Daemon does all the heavy lifting — building images, running containers, managing networks and volumes.

Figure 1 — Docker Architecture: Client, Host, and Registry

1Docker CLI receives the command and sends it to the Docker Daemon via REST API.
2Daemon checks if the requested image exists locally.
3If not found, the Daemon pulls the image from Docker Hub layer by layer.
4Daemon creates a new container from the image.
5Daemon starts the container — the process runs, then exits.
6Output is streamed back to your terminal.
03 Core Concept

Containers vs Virtual Machines

Both VMs and containers provide isolation, but they do it at very different levels. Understanding this difference is key to knowing when to use Docker and why it is so much faster.

Mental Model A VM is like renting an entire house — own kitchen, bedroom, bathroom. A container is like renting a hotel room. You share the building's plumbing and electricity, but your room is completely private.

04 Images

Docker Images and the Layer System

A Docker image is a read-only, immutable template used to create containers. Think of it as a class in object-oriented programming — you can create many instances (containers) from one class (image). Images are built from Dockerfiles and stored in registries.

Every instruction in a Dockerfile creates a new read-only layer. Layers are cached and reused across builds — if a layer has not changed, Docker skips rebuilding it. This makes builds dramatically faster. When you run a container, Docker adds a thin writable layer on top.

Writable Container Layer R/W
APP Layer   COPY . /app R/O
Deps Layer   RUN npm install R/O
Runtime Layer   FROM node:18 R/O
OS Base Layer   ubuntu:22.04 R/O

Figure 3 — Image Layer Stack: OS at bottom, writable container layer at top

Layer Facts

  • Layers are identified by a SHA-256 content hash
  • Shared layers are stored only once on disk
  • Pulling an image only downloads missing layers
  • All image layers except the top are read-only
  • docker image history shows all layers and sizes

Layer Best Practices

  • Never store secrets in Dockerfile or image layers
  • Minimize layers — combine RUN commands with &&
  • Order instructions from least to most frequently changed
  • Use .dockerignore to exclude files from build context
  • Base images like alpine are tiny — only about 5 MB

05 Dockerfile

Writing a Dockerfile

A Dockerfile is a plain text file containing step-by-step instructions that Docker follows to build an image. Each instruction becomes a layer in the image. The file name must be exactly Dockerfile.

All Dockerfile Instructions

InstructionPurposeExample
FROMSets the base image — must be first instructionFROM ubuntu:22.04
RUNExecutes commands during build, creates a new layerRUN apt-get install -y curl
CMDDefault command when container starts (overridable)CMD ["node", "app.js"]
ENTRYPOINTFixed command that always runs; CMD becomes its argsENTRYPOINT ["/entrypoint.sh"]
COPYCopies files from build context into imageCOPY ./src /app/src
ADDLike COPY but can extract .tar and fetch URLsADD app.tar.gz /app
WORKDIRSets working directory for subsequent instructionsWORKDIR /usr/src/app
EXPOSEDocuments which ports the container listens onEXPOSE 8080
ENVSets an environment variable in the containerENV NODE_ENV=production
ARGBuild-time variable (not available at runtime)ARG VERSION=1.0
VOLUMECreates a mount point for persistent storageVOLUME ["/data"]
USERSets user for RUN, CMD, ENTRYPOINTUSER appuser
HEALTHCHECKTells Docker how to test container healthHEALTHCHECK CMD curl -f http://localhost/
LABELAdds metadata key-value pairs to the imageLABEL maintainer="you@example.com"

CMD vs ENTRYPOINT

CMD — Overridable Default

  • Sets the default command to run
  • Can be completely overridden at runtime
  • docker run myimage /bin/bash replaces it
CMD ["node", "app.js"]

ENTRYPOINT — Fixed Executable

  • Sets the executable that always runs
  • CMD becomes its default arguments
  • Only --entrypoint flag can override it
ENTRYPOINT ["nginx", "-g"]
CMD ["daemon off;"]
Golden Rule Use ENTRYPOINT when you want the container to always behave like a specific executable. Use CMD when you want a default that users might want to override.

Real-World Node.js Dockerfile

Dockerfile — Node.js Production # Stage 1: Build
FROM node:18-alpine AS builder
WORKDIR /usr/src/app

# Install dependencies first (cache-friendly layer)
COPY package*.json ./
RUN npm ci --only=production

# Stage 2: Production image
FROM node:18-alpine
WORKDIR /usr/src/app

# Copy only what we need from builder
COPY --from=builder /usr/src/app/node_modules ./node_modules
COPY . .

# Create non-root user for security
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

EXPOSE 3000
HEALTHCHECK --interval=30s --timeout=5s \
  CMD wget -qO- http://localhost:3000/health || exit 1

CMD ["node", "server.js"]

07 Data Persistence

Docker Volumes

Critical Concept Container filesystems are ephemeral. Any data written inside a container is lost when the container is removed. Volumes solve this — they live outside the container lifecycle.

Figure 4 — Volume mount: persistent host storage connected to container filesystem

Three Types of Docker Storage

Named Volumes

Managed entirely by Docker. Stored in /var/lib/docker/volumes/. Best for databases. Easy to backup and restore. Syntax: -v my_vol:/container/path

Bind Mounts

Mounts a specific host directory. Full access to host filesystem. Perfect for live code reloading in dev. Depends on host structure. Syntax: -v /host/path:/container/path

tmpfs Mounts

Stored in host memory only (RAM). Never written to disk. Useful for sensitive temporary data. Data is gone when the container stops. Syntax: --tmpfs /tmp

Volume Commands # Create and use a named volume with Postgres
docker volume create my_database
docker run -d --name postgres_db \
  -e POSTGRES_PASSWORD=secret \
  -v my_database:/var/lib/postgresql/data \
  postgres:15

# Inspect, remove, backup
docker volume ls
docker volume inspect my_database
docker volume rm my_database
docker volume prune

# Backup a volume to a tar file
docker run --rm -v my_database:/data -v $(pwd):/backup \
  ubuntu tar czf /backup/db_backup.tar.gz /data
08 Networking

Docker Networking

Docker creates virtual networks that allow containers to communicate securely. Each container gets its own network namespace — its own virtual IP, ports, and routing table. You control exactly what can reach what.

Figure 5 — The four Docker network driver modes

bridge (Default)

Docker creates a private network via the docker0 bridge. Containers get their own IP. Use -p to map ports to the host. Containers on the same bridge can talk to each other.

host

Container shares the host's network stack directly — no network isolation. Highest performance (no NAT overhead). Container uses host's IP and ports directly. Use only when performance is critical.

none

Container has no network access whatsoever. Complete network isolation. Good for batch processing jobs, security-sensitive workloads, or testing.

overlay (Swarm)

Connects containers across multiple Docker hosts using VxLAN tunneling. Required for Docker Swarm services. Provides encrypted cross-host communication.

Pro Tip — Container DNS On custom networks (not the default bridge), Docker runs an embedded DNS server. Containers can reach each other by name instead of IP address. Your app can use http://database:5432 instead of hardcoding IPs.

09 Multi-container Apps

Docker Compose

Real-world apps are not single containers. A typical web app needs a web server, a database, a cache, and a message queue. Docker Compose lets you define all of these in one docker-compose.yml file and start them all with one command.

docker-compose.yml — Full-stack Web App version: "3.9"
services:
  web:
    build: .
    ports:
      - "8000:8000"
    environment:
      - DATABASE_URL=postgresql://user:pass@db:5432/mydb
      - REDIS_URL=redis://cache:6379
    volumes:
      - ./src:/app/src # live reload in dev
    depends_on: [db, cache]
    networks: [app_network]

  db:
    image: postgres:15
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: mydb
    volumes:
      - db_data:/var/lib/postgresql/data
    networks: [app_network]

  cache:
    image: redis:7-alpine
    networks: [app_network]

volumes:
  db_data:
networks:
  app_network:
    driver: bridge

10 Orchestration

Docker Swarm

Docker Swarm is Docker's native clustering and orchestration tool. It turns a group of Docker hosts into a single virtual Docker host. Swarm handles load balancing, rolling updates, health checks, and automatic rescheduling of failed services.

Manager Node
Manager
Assigns tasks
Load balances
Manages state
→
Worker Node 1Container A   Container B
Worker Node 2Container C   Container D
Worker Node 3Container E   (standby)

Figure 6 — Swarm cluster: one manager, multiple worker nodes

Key Concepts

  • Manager Nodes control the swarm and assign tasks to workers
  • Worker Nodes execute the container tasks
  • Services define the desired state of a set of containers
  • Tasks are single containers running on a node
  • Built-in load balancing across all replicas
  • Automatic failover — if a node dies, tasks reschedule

When to Use Swarm

  • Works best when you are already on Docker
  • Simpler to set up than Kubernetes
  • Supports rolling updates with zero downtime
  • Uses overlay networks for cross-host communication
  • Manages secrets and configs securely
  • Less powerful than Kubernetes for very large clusters
Swarm Setup and Service Management # Initialize the swarm on the manager node
docker swarm init --advertise-addr 192.168.1.100

# Join as worker (use token from init output)
docker swarm join --token SWMTKN-1-... 192.168.1.100:2377

# Create a service with 3 replicas
docker service create --name my_web --replicas 3 --publish 80:80 nginx:latest

# Scale up
docker service scale my_web=5

# Rolling update with zero downtime
docker service update --image nginx:1.25 my_web

# View nodes and services
docker node ls
docker service ls
11 Debugging

Debugging and Monitoring Containers

Debugging Toolkit # View container output (last 50 lines, past hour)
docker logs --tail=50 --since=1h myapp

# Get a shell inside a running container
docker exec -it myapp bash
docker exec -it myapp sh # for alpine-based images

# Real-time CPU, memory, network, disk I/O stats
docker stats
docker stats --format "table {{.Name}}\t{{.CPUPerc}}\t{{.MemUsage}}"

# Full JSON config, network, volumes, state
docker inspect myapp
docker inspect -f "{{.State.Status}}" myapp

# List processes running inside the container
docker top myapp

# Stream real-time events from the Docker daemon
docker events --filter type=container
Debug Build Failures When a build fails: find the last successful layer ID in the build output, then run docker run -it <layer_id> /bin/sh — this drops you inside the last-good state of the image so you can debug interactively.
12 Security

Security Best Practices

Never Run as Root

Always add a non-root user in your Dockerfile. Use the USER instruction to switch. Root in a container can be root on the host if misconfigured.

Never Put Secrets in Images

ENV, ARG, and COPY all leave traces in layers. Use Docker Secrets, env files, or a secrets manager (Vault, AWS Secrets Manager) at runtime.

Use Minimal Base Images

Start from alpine, distroless, or scratch. Fewer packages means a smaller attack surface. A 5 MB Alpine image has far fewer vulnerabilities than a 900 MB Ubuntu image.

Use Multi-stage Builds

Build tools should not be in production images. Multi-stage builds let you compile in one stage and copy only the binary to the final image.

Scan for Vulnerabilities

Use docker scout, Trivy, Snyk, or AWS ECR scanning to detect known CVEs before deployment.

Limit Capabilities

Use --cap-drop ALL and --cap-add only what you need. Prevents privilege escalation attacks.

Secure Dockerfile — Python FROM python:3.11-slim AS builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

FROM python:3.11-slim
WORKDIR /app
# Copy only app and dependencies, not build tools
COPY --from=builder /usr/local/lib/python3.11 /usr/local/lib/python3.11
COPY . .

# Create and use non-root user
RUN adduser --disabled-password --gecos "" appuser \
    && chown -R appuser:appuser /app
USER appuser

# Drop all capabilities except what is needed (set at runtime)
# --cap-drop ALL --cap-add NET_BIND_SERVICE
# Read-only filesystem: --read-only --tmpfs /tmp

EXPOSE 8080
CMD ["python", "app.py"]

Key Takeaways — Remember These Forever Containers are isolated processes sharing the host OS kernel — fast and lightweight. Images are read-only layered blueprints — immutable, shareable, versioned. A Dockerfile is the recipe for building an image — FROM, then RUN, then CMD. Volumes are the only way to persist data beyond container lifetime. Networks are isolated virtual environments — use custom networks for container DNS. Compose manages multi-container apps with one YAML file. On security: never run as root, never embed secrets, always use minimal images.
